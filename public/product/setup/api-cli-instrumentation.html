Product Intelligence

Track API, CLI & Agent Usage

Understand how customers engage with your API, CLI, and AI agents — not just whether they work, but how they're actually being used.

This is about engagement, not performance

ThriveStack is not an API monitoring or observability tool. We don't track latency, error rates, or uptime — tools like Treblle or Moesif do that. ThriveStack tracks who is using your API or CLI, which commands and endpoints they reach for, and how that usage patterns across accounts over time.

Why track this?

When your product exposes an API, CLI, or AI agent, a large portion of customer value is delivered outside a browser. That usage is invisible to traditional product analytics — you can't install a JavaScript snippet in a terminal session or an API call.

Instrumenting these surfaces lets you answer questions like:

  • Which API endpoints or CLI commands do power users rely on most?
  • Which accounts have adopted your API but never touched the dashboard?
  • Is a specific feature being used through the API but ignored in the UI?
  • Which companies are deeply embedded vs. barely active?

The event

Every API, CLI, or agent interaction is tracked as a single feature_used event. Two properties distinguish the surface:

Property Description Example values
interface The surface through which the customer interacted api  ·  cli  ·  agent
feature_name The specific endpoint, command, or tool they invoked orders  ·  list  ·  summarize

The full event shape: 

{
  "event_name": "feature_used",
  "user_id": "usr_k9mxbt32lw",  // user's unique identifier — or the ref id of their API key / token
  "timestamp": "2026-04-30T07:08:47.079Z",
  "context": {
    "group_id": "acc_p7nqz84yhe"  // account's unique identifier
  },
  "properties": {
    "interface": "api",       // api | cli | agent
    "feature_name": "orders"  // the endpoint, command, or tool invoked
  }
}

user_id is the unique identifier for the user making the call — or the reference ID of the API key or token they authenticated with. group_id is the account's unique identifier. Both are required for account-level analytics to work.

How to send the event

curl --location 'https://api.app.thrivestack.ai/api/track' \
  --header 'accept: */*' \
  --header 'content-type: application/json' \
  --header 'x-api-key: YOUR_API_KEY' \
  --data '[
    {
      "event_name": "feature_used",
      "user_id": "usr_k9mxbt32lw",
      "timestamp": "2026-04-30T07:08:47.079Z",
      "context": {
        "group_id": "acc_p7nqz84yhe"
      },
      "properties": {
        "interface": "api",
        "feature_name": "orders"
      }
    }
  ]'

You can find your API key on the ThriveStack telemetry onboarding page.

What you get in ThriveStack

Once events flow in, ThriveStack correlates them with the rest of the account journey:

  • Feature usage breakdown by interface (API vs. CLI vs. agent) per account
  • Which commands or endpoints drive the most engagement
  • Accounts that are API-only — potential power users or at-risk accounts with no UI engagement
  • Usage trends over time, tied to the full bow-tie funnel (acquisition → activation → expansion)
